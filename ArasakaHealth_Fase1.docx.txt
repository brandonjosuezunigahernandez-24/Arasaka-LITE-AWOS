B. Problema, Objetivo y Alcance
B.1 Contexto del Problema
Los pacientes con diabetes mellitus requieren seguimiento continuo de su estado de salud y una comunicación fluida con su equipo médico. En la práctica cotidiana, la gestión de citas y el manejo del historial clínico se realizan de forma manual —agendas de papel, llamadas telefónicas o mensajes informales— lo que genera errores, olvidos y pérdida de información clínica relevante.


ArasakaHealth API propone una solución web centralizada que permita a médicos y pacientes gestionar perfiles clínicos y citas de forma ordenada, trazable y segura, con autenticación por roles y acceso diferenciado a la información.


B.2 Objetivo General
Desarrollar una API REST con Laravel 11 y Eloquent ORM que permita gestionar usuarios con roles clínicos (médico y paciente), sus perfiles de salud y el agendamiento de citas médicas, exponiendo los datos de forma segura y estandarizada.


B.3 Objetivos Específicos
1. Implementar autenticación segura con Laravel Sanctum (registro, inicio y cierre de sesión) con control de acceso por rol (doctor / patient).
2. Exponer endpoints CRUD para la gestión de perfiles de pacientes, incluyendo datos clínicos básicos como tipo de sangre, peso y estatura.
3. Permitir el agendamiento, listado y cancelación de citas médicas con validación de reglas de negocio (fecha futura, vínculo médico-paciente).
4. Estandarizar todas las respuestas JSON bajo una estructura uniforme y manejar correctamente los códigos de estado HTTP.
5. Documentar todos los endpoints con ejemplos de request/response y validar su funcionamiento con una colección de pruebas en Postman.


B.4 Alcance — Qué Incluye
* Módulo de autenticación: registro, login y logout con tokens Sanctum.
* Módulo de pacientes: CRUD de perfiles con datos clínicos básicos (nombre, CURP, tipo de sangre, peso, estatura).
* Módulo de citas: agendar, listar y cancelar citas entre médico y paciente vinculado.
* Control de acceso por rol en cada endpoint (doctor vs patient).
* Respuestas JSON estandarizadas con estructura { success, message, data }.
* Manejo correcto de códigos HTTP: 200, 201, 204, 400, 401, 403, 404, 422.
* Colección de pruebas en Postman con mínimo 10 requests (incluye casos de error).
* README con instrucciones de instalación, ejecución y ejemplos de uso.


B.5 Fuera de Alcance — Qué No Incluye
* Integración con sensor Dexcom CGM ni lecturas de glucosa en tiempo real.
* Chat o mensajería entre usuarios.
* Notificaciones push o en tiempo real (WebSockets).
* Panel de administración de usuarios.
* App móvil — la API es exclusivamente consumida desde web.
* Pagos, facturación ni expediente clínico electrónico oficial.

C.2 Requerimientos No Funcionales
ID
	Categoría
	Descripción
	RNF01
	Seguridad — Auth
	Todos los endpoints excepto /register y /login deben requerir token Bearer válido emitido por Sanctum. Sin token válido se retorna 401.
	RNF02
	Seguridad — Autorización
	El acceso a recursos se verifica por rol. Un paciente no puede ver datos de otro paciente ni confirmar citas; un médico no puede cancelar citas ajenas.
	RNF03
	Seguridad — Validación
	Todos los campos de entrada deben validarse con Form Requests de Laravel. Datos inválidos retornan 422 con el detalle de cada campo fallido.
	RNF04
	Manejo de Errores
	La API retorna códigos HTTP semánticamente correctos: 200, 201, 204, 401, 403, 404, 422. Nunca retorna 200 con error en el cuerpo.
	RNF05
	Estandarización JSON
	Todas las respuestas siguen la estructura: { success: bool, message: string, data: object|array } para éxito, y { success: false, message, errors } para errores.
	RNF06
	ORM
	El acceso a la base de datos se realiza exclusivamente a través de Eloquent ORM. No se permite SQL crudo en los controladores.
	RNF07
	Control de versiones
	El código fuente se gestiona en GitHub con ramas por módulo (feature/auth, feature/patients, feature/appointments).
	RNF08
	Documentación
	Todos los endpoints están documentados en el README con ejemplos de request y response. La colección Postman debe ser exportable (.json).

D. Modelo de Datos y Reglas de Negocio
D.1 Entidades y Atributos
Entidad
	Atributos
	Reglas de negocio
	User
	id (PK, autoincrement), name, email, password (hashed), role (doctor|patient), timestamps
	Email único. Role requerido, solo valores del enum. Contraseña hasheada con bcrypt, nunca en texto plano.
	PatientProfile
	id (PK), user_id (FK → users, 1:1), curp, blood_type, weight_kg, height_cm, phone, timestamps
	Solo existe si role = patient. CURP único de 18 caracteres. blood_type: enum A+|A-|B+|B-|AB+|AB-|O+|O-. weight_kg y height_cm deben ser positivos.
	Appointment
	id (PK), patient_id (FK → users), doctor_id (FK → users), scheduled_at (datetime), status (pending|confirmed|cancelled), notes, timestamps
	scheduled_at debe ser fecha futura al crear. Solo el paciente puede cancelar su propia cita. Solo el médico puede confirmar. Un mismo paciente no puede tener dos citas con el mismo médico en la misma hora.
	

D.2 Relaciones
* User 1:1 PatientProfile — un usuario de tipo patient tiene exactamente un perfil clínico extendido.
* User (patient) 1:N Appointment — un paciente puede tener múltiples citas.
* User (doctor) 1:N Appointment — un médico puede tener múltiples citas asignadas.

E. Historias de Usuario
#
	Historia de Usuario
	Criterios de Aceptación
	HU01
	Como usuario nuevo, quiero registrarme con nombre, email, contraseña y rol para acceder a la plataforma.
	✓ Registro exitoso retorna 201 con token y perfil.✓ Email duplicado retorna 422 con mensaje claro.✓ Contraseña menor a 8 chars retorna 422.
	HU02
	Como usuario registrado, quiero iniciar sesión con email y contraseña para obtener mi token de acceso.
	✓ Login correcto retorna 200 con token Bearer y objeto user.✓ Credenciales incorrectas retornan 401.✓ Campos vacíos retornan 422.
	HU03
	Como usuario autenticado, quiero cerrar sesión para que mi token quede invalidado.
	✓ Logout exitoso retorna 200 con mensaje de confirmación.✓ Token invalidado — uso posterior retorna 401.✓ Sin token en header retorna 401.
	HU04
	Como médico, quiero listar todos los pacientes registrados para elegir a quién agendarle una cita.
	✓ GET retorna 200 con arreglo de pacientes (nombre, email, perfil clínico).✓ Si no hay pacientes retorna arreglo vacío, no 404.✓ Un paciente que intente este endpoint recibe 403.
	HU05
	Como paciente, quiero actualizar mis datos clínicos (tipo de sangre, peso, estatura) para mantener mi perfil al día.
	✓ PUT exitoso retorna 200 con perfil actualizado.✓ Valores fuera de rango retornan 422.✓ Un paciente no puede editar el perfil de otro (403).
	HU06
	Como paciente, quiero agendar una cita con un médico seleccionando fecha y hora para organizar mi consulta.
	✓ POST exitoso retorna 201 con la cita en estado pending.✓ Fecha pasada retorna 422 con mensaje de error.✓ Doctor ID inexistente retorna 422.
	HU07
	Como paciente, quiero listar mis citas para conocer su estado actual.
	✓ GET retorna 200 con mis citas ordenadas por fecha.✓ Incluye nombre del médico y estado de cada cita.✓ No retorna citas de otros pacientes.
	HU08
	Como médico, quiero confirmar una cita pendiente de un paciente para validar mi disponibilidad.
	✓ PATCH exitoso retorna 200 con cita en estado confirmed.✓ Un paciente que intente confirmar recibe 403.✓ Cita inexistente retorna 404.

H.2 Tres Razones para Elegir Laravel
1. Estructura propia del framework — evidencia inmediata
Laravel 11 impone una arquitectura clara desde el inicio: carpetas routes/, app/Http/Controllers/, app/Http/Requests/, app/Models/ y app/Policies/. Esto cumple directamente con el criterio de validación de la práctica: 'se ve el framework en la estructura del proyecto'. Con un solo comando php artisan make:controller el framework genera los archivos en las carpetas correctas, y el archivo routes/api.php centraliza el registro de todos los endpoints.


2. Sanctum + Form Requests = seguridad y validación sin librerías extra
Laravel Sanctum provee autenticación stateless por token Bearer en minutos, sin configuración compleja. Los Form Requests permiten centralizar todas las reglas de validación (required, email, min:8, in:doctor,patient, date|after:now) fuera de los controladores, retornando automáticamente respuestas 422 con el detalle de cada campo inválido. Esto cubre los RNF de seguridad y manejo de errores sin una sola línea de código adicional.


3. Eloquent ORM — relaciones limpias sobre el modelo de datos existente
Las tres entidades del proyecto (User, PatientProfile, Appointment) tienen relaciones directas que Eloquent maneja con hasOne, hasMany y belongsTo. Esto permite escribir consultas como $doctor->appointments()->where('status','pending')->get() en lugar de SQL crudo, haciendo el código más legible, mantenible y alineado con las buenas prácticas de un framework real.

H.4 Riesgos y Mitigación
Riesgo
	Mitigación
	Equipo acostumbrado a JavaScript/TypeScript, PHP puede resultar diferente en sintaxis.
	La estructura de Laravel (rutas, controladores, middleware) es conceptualmente idéntica a Express.js. Se usará la documentación oficial laravel.com y el scaffolding de Artisan para reducir el tiempo de adaptación.
	Configuración del entorno PHP en las máquinas de los tres integrantes.
	Uso de Laravel Herd (macOS/Windows, instalador gráfico) o Docker con imagen php:8.3-fpm + composer para estandarizar el entorno. El comando php artisan serve levanta el servidor sin configuración de Apache/Nginx.
	Posibles conflictos de versiones entre PHP y paquetes de Composer.
	Laravel 11 requiere PHP 8.3+. Se documentará la versión exacta en el README y se usará composer install —no-dev para entornos de producción/pruebas.
